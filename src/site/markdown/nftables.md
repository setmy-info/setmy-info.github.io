# nftables

## Information

`nftables` is the modern packet filtering and classification framework used on many Linux systems. It replaces much of
the older `iptables`, `ip6tables`, `arptables`, and `ebtables` toolchain with one more consistent rules engine and one
configuration model.

In practical infrastructure and platform work, `nftables` is useful when you want to:

* control ingress and egress traffic directly at host level,
* build precise allowlist or denylist rules,
* manage `IPv4` and `IPv6` policy in one place,
* create `NAT`, forwarding, and router-like behavior,
* implement lower-level firewall policy without depending only on a higher abstraction layer.

For `SMI`-style operational usage, `nftables` is especially valuable when you need explicit, reviewable firewall logic
and want predictable rule order, reusable sets, and a cleaner model than legacy rule commands.

### Main functionalities and features

* **Tables, chains, and rules**: structured way to organize packet filtering logic,
* **Unified filtering model**: one framework for `IPv4`, `IPv6`, `NAT`, bridges, and more,
* **Sets and maps**: efficient allowlists and blocklists for addresses, ports, and networks,
* **Stateful filtering**: allow established connections while restricting new ones,
* **`NAT` support**: source `NAT`, masquerading, and destination `NAT`,
* **Logging and counters**: inspect what is being matched and dropped,
* **Atomic ruleset replacement**: load a complete ruleset in a consistent way.

### Typical use cases

* allow only `SSH` from a management subnet,
* expose `HTTP` and `HTTPS` for an application while dropping other inbound traffic,
* allow only selected outbound destinations or ports,
* build a host that performs `NAT` for an internal network,
* protect container, `VM`, or service hosts with explicit ingress and egress rules,
* complement tools such as `Squid` by preventing direct internet access except through approved paths.

## Core concepts

### Tables

A table is a top-level container for firewall logic.

Common table families include:

* `ip`: `IPv4` only,
* `ip6`: `IPv6` only,
* `inet`: combined `IPv4` and `IPv6`,
* `arp`: `ARP`,
* `bridge`: bridge traffic,
* `netdev`: early ingress filtering on interfaces.

For many server use cases, `inet` is a good default because it lets you express one rule set for both `IPv4` and
`IPv6` traffic.

### Chains

Chains contain rules.

Base chains are attached to a packet-processing hook such as:

* `input`: traffic destined for the local host,
* `output`: traffic generated by the local host,
* `forward`: traffic routed through the host,
* `prerouting`: before routing decision,
* `postrouting`: after routing decision.

When defining a base chain, you usually specify:

* the hook,
* a priority,
* a default policy such as `accept` or `drop`.

### Rules

Rules match packet attributes and then take an action.

Common matches include:

* protocol such as `tcp`, `udp`, `icmp`,
* source or destination address,
* source or destination port,
* connection state,
* input or output interface.

Common actions include:

* `accept`,
* `drop`,
* `reject`,
* `log`,
* `jump` to another chain,
* `masquerade`,
* `dnat`,
* `snat`.

### Connection tracking

State tracking is one of the most important operational concepts.

Typical firewall policy allows:

* `established` traffic,
* `related` traffic,
* selected new inbound traffic,
* selected outbound traffic.

This avoids writing symmetric allow rules for every reply packet.

### Sets and maps

Sets are very practical in real operations.

Use them when you want to maintain:

* approved source subnets,
* blocked addresses,
* allowed destination ports,
* allowed destination networks.

Instead of repeating many rules, you can match against one reusable set, which keeps the ruleset shorter and easier to
review.

### Rule order matters

Like most firewall systems, `nftables` evaluates rules in order inside a chain.

Operationally, this means:

* allow specific required traffic before broad drops,
* log selectively before dropping when you need troubleshooting,
* keep the rule flow readable and intentional,
* avoid mixing ad-hoc one-off rules with a managed baseline unless you understand the full ruleset.

## Installation

### Rocky Linux / AlmaLinux / CentOS / Fedora

On many modern systems `nftables` is already available or installed.

```bash
sudo dnf install -y nftables
sudo systemctl enable --now nftables
sudo systemctl status nftables
```

### Debian / Ubuntu

```bash
sudo apt update
sudo apt install -y nftables
sudo systemctl enable --now nftables
sudo systemctl status nftables
```

### Check version and service state

```bash
nft --version
sudo systemctl status nftables
```

Common configuration file:

```text
/etc/nftables.conf
```

## Basic commands

### Show current ruleset

```bash
sudo nft list ruleset
```

### Validate a ruleset file

Validate before applying changes:

```bash
sudo nft -c -f /etc/nftables.conf
```

### Load a ruleset file

```bash
sudo nft -f /etc/nftables.conf
```

### Flush current ruleset

Use this carefully, especially on remote servers.

```bash
sudo nft flush ruleset
```

### Save service-managed configuration

Typical persistent workflow is:

1. edit `/etc/nftables.conf`,
2. validate with `nft -c -f`,
3. load it with `nft -f`,
4. restart or enable the service if needed.

## Basic setup

### Minimal host firewall example

This example allows loopback, established traffic, and inbound `SSH`, `HTTP`, and `HTTPS`, while dropping other
unsolicited inbound traffic.

```conf
#!/usr/sbin/nft -f

flush ruleset

table inet filter {
    chain input {
        type filter hook input priority 0;
        policy drop;

        iif "lo" accept
        ct state established,related accept

        tcp dport { 22, 80, 443 } accept
        ip protocol icmp accept
        ip6 nexthdr ipv6-icmp accept
    }

    chain forward {
        type filter hook forward priority 0;
        policy drop;
    }

    chain output {
        type filter hook output priority 0;
        policy accept;
    }
}
```

This is a good teaching baseline, but production rules usually need more source restrictions, logging, and explicit
egress design.

### Safer `SSH` example limited to a management subnet

```conf
table inet filter {
    chain input {
        type filter hook input priority 0;
        policy drop;

        iif "lo" accept
        ct state established,related accept

        ip saddr 10.20.30.0/24 tcp dport 22 accept
        tcp dport { 80, 443 } accept
    }
}
```

This is often better than exposing `SSH` from everywhere.

## Restrict outbound access

### Basic egress control example

For many servers, outbound traffic is left fully open. In controlled environments you may want the opposite: allow only
required destinations or ports.

```conf
table inet filter {
    chain output {
        type filter hook output priority 0;
        policy drop;

        oif "lo" accept
        ct state established,related accept

        udp dport 53 accept
        tcp dport 53 accept
        tcp dport 443 accept
        tcp dport 80 accept
    }
}
```

This still allows broad web access. It is only a first step toward stronger egress control.

### More controlled egress for proxy-only internet access

If `Squid` is your approved egress path, the stronger pattern is:

* internal application hosts cannot reach the public internet directly,
* they can reach only internal infrastructure such as `DNS`, package mirrors, and the proxy,
* the proxy host itself has the more specific outbound allowlist.

On an application host, you might allow only `DNS` plus the internal proxy:

```conf
table inet filter {
    chain output {
        type filter hook output priority 0;
        policy drop;

        oif "lo" accept
        ct state established,related accept

        ip daddr 10.10.10.53 udp dport 53 accept
        ip daddr 10.10.10.53 tcp dport 53 accept
        ip daddr 10.10.20.10 tcp dport 3128 accept
    }
}
```

This helps enforce that tools use the proxy instead of opening direct external connections.

## Example: allow only selected external networks from a proxy host

At host firewall level, `nftables` does not understand domain names dynamically in the same way a proxy policy does.

That means:

* it is good for network-level restrictions,
* it is not ideal for application-level hostname policy,
* strict control of `OpenAI` or `Anthropic` by hostname is usually better done in `Squid` or another proxy.

Still, if you know a stable destination subnet or internal upstream path, `nftables` can restrict traffic to those
addresses.

Example with a set:

```conf
table inet filter {
    set allowed_https_targets {
        type ipv4_addr
        elements = { 203.0.113.10, 203.0.113.11 }
    }

    chain output {
        type filter hook output priority 0;
        policy drop;

        ct state established,related accept
        ip daddr @allowed_https_targets tcp dport 443 accept
    }
}
```

Be careful with external provider address ranges because they often change.

### Example: default drop, allow one set, but still block selected special addresses

Sometimes the policy is not just "drop all, allow these addresses".

Sometimes you need this order:

1. default deny everything,
2. allow only one approved address set,
3. but still deny a few specific addresses inside that otherwise approved scope.

This is a common pattern when:

* one partner subnet is mostly trusted,
* a provider gave you a larger range but only some nodes should be reachable,
* you want an emergency blocklist that overrides a broader allowlist.

Because rule order matters, place the explicit deny before the broader allow.

```conf
table inet filter {
    set allowed_targets {
        type ipv4_addr
        elements = { 198.51.100.0/24, 203.0.113.10 }
    }

    set blocked_targets {
        type ipv4_addr
        elements = { 198.51.100.25, 198.51.100.99 }
    }

    chain output {
        type filter hook output priority 0;
        policy drop;

        oif "lo" accept
        ct state established,related accept

        ip daddr @blocked_targets tcp dport 443 counter drop
        ip daddr @allowed_targets tcp dport 443 counter accept
    }
}
```

Operational meaning:

* everything is denied by default because the chain policy is `drop`,
* only `HTTPS` to the addresses in `allowed_targets` can pass,
* even if an address belongs to that broader approved scope, anything in `blocked_targets` is still denied first.

This is a practical way to express "drop all, allow something, but still drop selected exceptions".

If you need subnet-based allow plus host-based deny, the pattern stays the same: put the more specific deny rule first,
then the
broader allow rule after it.

## `NAT` examples

### Masquerading for an internal subnet

If the host acts as a gateway for an internal network:

```conf
table ip nat {
    chain postrouting {
        type nat hook postrouting priority 100;

        ip saddr 192.168.50.0/24 oif "eth0" masquerade
    }
}
```

This rewrites outbound packets from the internal subnet to use the gateway host address.

In practice, `masquerade` alone is not enough. The host also needs forwarding enabled and a matching forward-policy.

Example for a small gateway where:

* `lan0` is the internal interface,
* `wan0` is the external interface,
* `192.168.50.0/24` is the internal subnet.

```conf
table inet filter {
    chain forward {
        type filter hook forward priority 0;
        policy drop;

        ct state established,related accept
        iif "lan0" oif "wan0" ip saddr 192.168.50.0/24 accept
    }
}

table ip nat {
    chain postrouting {
        type nat hook postrouting priority 100;

        ip saddr 192.168.50.0/24 oif "wan0" masquerade
    }
}
```

Also remember:

```bash
sudo sysctl -w net.ipv4.ip_forward=1
```

And make it persistent if this host is intended to operate as a router or `NAT` gateway.

### Destination `NAT` / port forwarding example

```conf
table ip nat {
    chain prerouting {
        type nat hook prerouting priority -100;

        tcp dport 8443 dnat to 192.168.50.20:443
    }
}
```

This forwards inbound traffic on `8443` to an internal service on `443`.

### Full port-forward `NAT` example with matching forward rules

Real `DNAT` setups usually need both:

* the `nat` rule that rewrites the destination,
* the `forward` rule that actually permits the forwarded traffic.

Example:

* public traffic arrives on `wan0`,
* the gateway forwards `8443` to `192.168.50.20:443` on `lan0`.

```conf
table inet filter {
    chain forward {
        type filter hook forward priority 0;
        policy drop;

        ct state established,related accept
        iif "wan0" oif "lan0" ip daddr 192.168.50.20 tcp dport 443 accept
    }
}

table ip nat {
    chain prerouting {
        type nat hook prerouting priority -100;

        iif "wan0" tcp dport 8443 dnat to 192.168.50.20:443
    }
}
```

If return traffic leaves through the same gateway toward the internet, you may also need a `postrouting` masquerade
rule,
depending on the routing design.

## Logging and troubleshooting

### Add logging before drop rules carefully

Example:

```conf
table inet filter {
    chain input {
        type filter hook input priority 0;
        policy drop;

        ct state established,related accept
        tcp dport 22 accept
        log prefix "nft-input-drop: " counter drop
    }
}
```

Do not log everything at high volume on busy systems unless you really need it, because logging can become noisy.

### Useful commands

```bash
sudo nft list ruleset
sudo nft list table inet filter
sudo nft monitor
sudo journalctl -u nftables -n 100 --no-pager
```

### Counter inspection

Counters help show whether a rule is actually matching:

```conf
tcp dport 443 counter accept
```

This is very helpful when diagnosing why traffic is not flowing as expected.

## Operational tips and tricks

### Always plan remote access safety

The easiest firewall outage is locking yourself out over `SSH`.

Before loading a new ruleset on a remote host:

* confirm the `SSH` allow rule is correct,
* keep an existing session open,
* validate with `nft -c -f`,
* apply changes carefully during a maintenance window when possible.

### Prefer complete managed rulesets over random one-off commands

For stable operations, keep a reviewed ruleset file under configuration management rather than depending only on many
manual interactive commands.

### Avoid mixing too many firewall managers on one host

Be careful when combining:

* raw `nft` commands,
* `firewalld`,
* container runtime firewall management,
* orchestration systems,
* cloud-init or bootstrap scripts.

If several tools write firewall state independently, troubleshooting becomes harder.

### `firewalld` versus raw `nftables`

Use `firewalld` when you want:

* a higher-level operational interface,
* zone-based administration,
* simpler day-to-day service opening.

Use raw `nftables` when you want:

* exact packet-level control,
* explicit low-level rules,
* custom set/map logic,
* minimal abstraction.

### Think about both ingress and egress

Teams often focus only on inbound protection. In controlled environments, outbound policy matters too.

Good operational questions include:

* which services may this host expose,
* which external destinations should it reach,
* should this host reach the internet directly at all,
* should traffic be forced through a proxy or gateway.

## Common issues

### Rules load, but traffic still does not work

Possible causes:

* wrong chain hook,
* wrong table family,
* missing `ct state established,related accept`,
* wrong interface name,
* another firewall manager overwrote the rules,
* routing issue rather than firewall issue.

Fix:

* inspect the loaded ruleset,
* add counters to important rules,
* verify interfaces with `ip addr`,
* confirm whether `firewalld`, `Docker`, `Podman`, or `Kubernetes` also manage firewall state.

### `NAT` works partly, but forwarding fails

Check not only `nftables`, but also:

* `net.ipv4.ip_forward=1`,
* reverse path filtering where relevant,
* correct forward-chain allow rules,
* upstream routing.

`NAT` alone does not guarantee packets can traverse the host.

### Hostname-based restrictions are harder than expected

This is a very common design mistake.

`nftables` is primarily network-layer filtering. If you need to allow `api.openai.com` but block unrelated hosts on
`443`, a proxy such as `Squid` is usually the better control point because it can inspect the requested destination
host at application/proxy level.

### `IPv6` is accidentally left open

If you write rules only in `ip` tables, `IPv6` traffic may follow a different path.

For many modern systems, prefer `inet` where appropriate so your policy covers both protocol families consistently.

## When developers and operators should know this

From a practical infrastructure and platform perspective:

* junior engineers should understand the difference between inbound and outbound filtering and know how to inspect the
  active ruleset,
* mid-level engineers should be able to read and apply a managed `nftables` policy, troubleshoot counters, and safely
  open or restrict services,
* senior engineers should be able to design layered firewall policy across hosts, proxies, `NAT`, containers, and
  orchestration systems.
